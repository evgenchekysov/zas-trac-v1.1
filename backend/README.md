ZAS‑TRAC | Backend Architecture
Назначение
Backend ZAS‑TRAC реализует:

бизнес‑правила работы с заявками и рабочим временем;
API для frontend и Telegram‑бота;
контроль допустимых действий пользователей в рамках бизнес‑логики.

Backend не отвечает за аутентификацию, роли и первичную безопасность данных —
эти функции вынесены на уровень Supabase (Auth + RLS).

Базовые архитектурные принципы
Идентификация и доступ
Единственный источник идентификации пользователя — Supabase Auth (auth.uid).
Backend:

не хранит пароли;
не выпускает собственные JWT;
не реализует RBAC.

Все ограничения доступа и прав реализуются на уровне базы данных
через PostgreSQL Row Level Security (RLS).
Backend не может обойти RLS даже при ошибке в коде.

Разделение ответственности
HTTP (FastAPI routers)
        ↓
Business Services (rules & invariants)
        ↓
CRUD / DB access
        ↓
PostgreSQL + RLS (Supabase)

Ключевые сервисы (Domain Layer)
ticket_service.py
Жизненный цикл заявок
Статусы:
NEW → ASSIGNED → IN_PROGRESS → PAUSED → DONE → CLOSED

Основные правила:

DONE — завершение работ в целом;
PAUSED — агрегированное состояние при отсутствии активных сессий;
CLOSED — финальное управленческое закрытие заявки.

UI‑контракт:

Начать работу
Остановить работу
Работы выполнены
Закрыть заявку

📌 Является архитектурным контрактом проекта.

Приоритет заявок (Ticket Priority)
Каждая заявка в ZAS‑TRAC имеет приоритет.
Приоритет:

является атрибутом заявки, а не статусом;
не участвует в state‑machine статусов;
не управляет сессиями и рабочим временем.

Уровни приоритета
Фиксированный набор:

EMERGENCY — аварийный (немедленная реакция);
HIGH — плановый высокий;
NORMAL — обычный (по умолчанию);
LONG_TERM — долгосрочный / факультативный.

Архитектурные ограничения
Приоритет НЕ может:

запускать или останавливать работу;
изменять агрегированный статус заявки;
обходить правила участия (participant / owner).

Использование
Приоритет применяется только для:

сортировки и фильтрации очереди заявок;
диспетчеризации;
уведомлений (Telegram);
будущих SLA, аналитики и отчётов.

Управление

задаётся при создании заявки;
любое изменение фиксируется в Audit / Event Log;
ограничения доступа:

worker не может повышать приоритет выше NORMAL;
admin / dispatcher может устанавливать любой уровень.



Контроль доступа реализуется через Supabase RLS, не в backend‑коде.
⚠️ Использование приоритета в расчёте статусов или управлении сессиями
считается архитектурной ошибкой.

session_service.py
Учёт фактического рабочего времени.
Инварианты:

одна активная сессия на пользователя;
сессии строго персональные;
автопауза при переключении задач;
DONE автоматически завершает активные сессии.


audit_service.py
Фиксация всех значимых событий:

создание заявок;
смена статусов;
запуск / остановка сессий;
автопаузы, DONE, CLOSED;
изменение приоритета.

Используется для:

отладки;
отчётов;
Telegram‑уведомлений.

Append‑only. Не влияет на бизнес‑логику.

notification_service.py
Абстрактный слой уведомлений.

каналы (Telegram, email и др.) реализуются отдельно;
не содержит бизнес‑логики;
реагирует на события из audit‑log.


asset_service.py
Связь оборудования с заявками.

реестр активов;
используется администраторами;
не участвует в статусной логике заявок.


Диагностические эндпоинты

GET /healthz — проверка сервиса
GET /diag/whoami — текущий пользователь (auth.uid)
GET /diag/db — проверка подключения к БД

Используются для:

Railway;
отладки;
Telegram‑интеграции.


Важные инварианты проекта

Backend не может нарушить RLS даже при ошибке в коде.
Сессии — персональные, статусы заявок — агрегированные.
DONE и PAUSED имеют принципиально разный смысл.
Архитектурные контракты первичны по отношению к реализации.
