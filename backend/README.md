ZAS‑TRAC | Backend Architecture
Назначение
Backend ZAS‑TRAC реализует:

бизнес‑правила работы с заявками и рабочим временем;
API для frontend и Telegram‑бота;
контроль допустимых действий пользователей.

Backend не отвечает за аутентификацию, роли и первичную безопасность данных —
эти функции вынесены на уровень Supabase (Auth + RLS).

Базовые архитектурные принципы
1. Идентификация и доступ

Единственный источник идентификации пользователя — Supabase Auth (auth.uid)
Backend:

не хранит пароли;
не выпускает собственные JWT;
не реализует RBAC.


Все ограничения доступа реализованы в базе данных через PostgreSQL RLS.


2. Разделение ответственности
HTTP (FastAPI routers)
        ↓
Business Services (rules & invariants)
        ↓
CRUD / DB access
        ↓
PostgreSQL + RLS (Supabase)

Ключевые сервисы (Domain Layer)
ticket_service.py

Жизненный цикл заявок
Статусы: NEW → ASSIGNED → IN_PROGRESS → PAUSED → DONE → CLOSED
Правила:

DONE — завершение работ в целом;
PAUSED — отсутствие активных сессий;
CLOSED — финальное управленческое закрытие.


UI‑контракт:

Начать работу
Остановить работу
Работы выполнены
Закрыть заявку



📌 Является архитектурным контрактом проекта

session_service.py

Учёт фактического рабочего времени
Инварианты:

одна активная сессия на пользователя;
сессии строго персональные;
автопауза при переключении задач;
DONE автоматически завершает активные сессии.




audit_service.py

Фиксация всех значимых событий:

создание заявок;
смена статусов;
запуск / остановка сессий;
автопаузы, DONE, CLOSED.


Используется для:

отладки;
отчётов;
Telegram‑уведомлений.




notification_service.py

Абстрактный слой уведомлений
Каналы (Telegram, email и др.) реализуются позже
Не влияет на бизнес‑логику


asset_service.py

Связь оборудования с заявками
Реестр активов
Используется администраторами


Диагностические эндпоинты

GET /healthz — проверка сервиса
GET /diag/whoami — текущий пользователь (auth.uid)
GET /diag/db — проверка подключения к БД

Используются для:

Railway;
отладки;
Telegram‑интеграции.


Важные инварианты проекта

Backend не может нарушить RLS даже при ошибке в коде.
Сессии — персональные, статусы заявок — агрегированные.
DONE и PAUSED имеют принципиально разный смысл.
Архитектурные контракты первичны по отношению к реализации.
