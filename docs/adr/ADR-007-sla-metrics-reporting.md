ADR‑007: SLA, Metrics & Reporting
Статус: Accepted (production baseline)
Дата: 2026‑04‑27
Контекст: ZAS‑TRAC Backend

1. Контекст
Система ZAS‑TRAC используется для диспетчеризации, исполнения и контроля заявок, где критически важны:

скорость реакции системы,
соблюдение сроков выполнения,
прозрачность ответственности,
управляемость отклонений.

Ранее в архитектуре зафиксированы:

ADR‑003 — жизненный цикл тикета,
ADR‑004 — события и audit,
ADR‑005 — workflow‑паттерн,
ADR‑006 — рабочие сессии и фактическое время.

Необходимо формально зафиксировать:

модель SLA,
перечень рассчитываемых метрик,
источники данных,
правила фиксации и обработки нарушений SLA.


2. Определение SLA
SLA (Service Level Agreement) — это обязательство системы обеспечить обработку тикета в заданные временные рамки, подтверждаемое фактическими данными, зафиксированными в системе.
В ZAS‑TRAC SLA является:

✅ обязательным,
✅ вычисляемым,
✅ воспроизводимым,
❌ не ручным,
❌ не редактируемым.

SLA всегда рассчитывается, даже если он нарушен.

3. Принципиальное решение

Нарушение SLA не является ошибкой системы.
Нарушение SLA является фактом и фиксируется отдельным событием.

SLA не управляет жизненным циклом тикета напрямую и не изменяет его состояние.

4. Типы SLA в ZAS‑TRAC
Вводится следующий минимальный набор SLA‑метрик.

4.1 Response SLA — SLA реакции
Response SLA отражает скорость реакции системы на поступление заявки.
Реакция может произойти двумя равноправными способами:

диспетчер назначил исполнителя,
исполнитель самостоятельно взял заявку в работу.

Формула Response SLA
Response SLA =
min(
  ticket_created → ticket_assigned,
  ticket_created → ticket_started
)

Считается тот интервал, который завершился раньше.
Смысл
Response SLA показывает, когда система отреагировала на заявку, а не когда начались работы.
Осознанно НЕ включено
Интервал:
ticket_assigned → ticket_started

не является SLA‑метрикой, так как может быть сознательно длинным
(плановые, низкоприоритетные, отложенные задачи).

4.2 Resolution SLA — SLA выполнения работ
Resolution SLA отражает скорость непосредственного выполнения работ.
Формула
ticket_started → ticket_done

Источник данных:

события ticket_started, ticket_done
рабочие сессии (ADR‑006)


4.3 Closure SLA — SLA административного закрытия
Closure SLA отражает скорость реакции диспетчера / администратора
на завершённую работу.
Формула
ticket_done → ticket_closed

Смысл:

контроль административного контура,
выявление узких мест на этапе проверки и закрытия.


5. Источники данных (жёсткое правило)

Данные                Источник
Создание тикета       ticket_created event
Назначение            ticket_assigned event
Начало работ          ticket_started event
Завершение работ      ticket_done event
Закрытие              ticket_closed event
Фактическая работа    work sessions (ADR‑006)

❌ UI
❌ ручные поля
❌ внешние системы
Audit‑события и рабочие сессии являются единственным источником истины.

6. Пороги SLA
6.1 Кто задаёт пороги
Пороги SLA:

являются конфигурацией системы,
устанавливаются SuperAdmin / System Admin,
не хардкодятся в бизнес‑логике.

Пороги могут зависеть от:

типа заявки,
приоритета,
категории,
эксплуатационной политики.

ADR‑007 фиксирует принцип, но не UI или формат хранения.

7. Нарушение SLA (sla_breached event)
При превышении SLA‑порога система обязана зафиксировать событие:
sla_breached

7.1 Когда создаётся sla_breached

в момент обнаружения превышения порога,
один раз для каждого типа SLA на тикет,
независимо от дальнейшей судьбы тикета.


7.2 Контракт sla_breached event
Минимальный набор данных:

ticket_id
sla_type (response | resolution | closure)
threshold
actual_value
breached_at
current_state
actor_context
metadata (опционально)


8. Уведомления и напоминания (важное разделение)
Ключевой принцип

sla_breached — это событие.
Повторные уведомления — это policy, а не новые события.

Поведение уведомлений

после sla_breached запускается notification‑policy,
напоминания могут повторяться через настраиваемые интервалы,
повторы прекращаются, когда:

заявка назначена (ticket_assigned), или
заявка взята в работу (ticket_started), или
соответствующий SLA больше не применим.



Почему так

audit остаётся чистым,
нет заспама событиями,
гибкая настройка без переписывания архитектуры.


9. Что МОЖНО делать с sla_breached
✅ допустимо:

уведомления (TG, email, dashboards),
управленческая эскалация,
аналитика и отчёты,
сигналы для BI,
пост‑фактум анализ процессов.


10. Что ЗАПРЕЩЕНО делать с sla_breached
❌ автоматически менять состояние тикета
❌ запускать workflow
❌ переоткрывать / закрывать тикет
❌ корректировать события или время
❌ реализовывать санкции на уровне логики
SLA breach — сигнал, а не команда.

11. Reporting & Metrics
Metrics

рассчитываются централизованно,
используют только audit + sessions,
не содержат бизнес‑логики.

Reports

read‑only представления,
не влияют на workflow,
не изменяют данные.


12. Инварианты ADR‑007

SLA всегда рассчитывается
SLA не редактируется вручную
Нарушение SLA фиксируется событием
sla_breached не изменяет state‑machine
Повторы — ответственность notification‑policy
Audit и sessions — единственный источник истины

Нарушение любого пункта считается архитектурной ошибкой.

13. Осознанно исключено
На текущем этапе:

ручная корректировка SLA,
частичное выполнение SLA,
скрытые коэффициенты,
автоматическое исправление истории,
SLA‑логика в UI.
