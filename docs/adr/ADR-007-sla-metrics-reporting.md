ADR‑007: SLA, Metrics & Reporting
Статус: Accepted (production baseline)  
Дата: 2026‑04‑27  
Контекст: ZAS‑TRAC Backend  

-------------------------------------------------
1. Контекст
-------------------------------------------------

Система ZAS‑TRAC используется для диспетчеризации, исполнения и контроля заявок,
где критически важны:

— скорость реакции;
— соблюдение сроков;
— прозрачность ответственности;
— управляемость отклонений.

В архитектуре ранее зафиксированы:

— ADR‑003 — жизненный цикл тикета  
— ADR‑004 — события и audit  
— ADR‑005 — workflow  
— ADR‑006 — рабочие сессии и учёт времени  

Данный ADR определяет:

— модель SLA;  
— базовые метрики системы;  
— источники данных;  
— правила фиксации SLA и нарушений;  
— принципы reporting.

-------------------------------------------------
2. Определение SLA
-------------------------------------------------

SLA (Service Level Agreement) — обязательство системы обеспечить
обработку тикета в заданные временные рамки,
подтверждаемое фактическими данными.

SLA в системе:

✅ всегда рассчитывается  
✅ основан на фактах (events + sessions)  
✅ воспроизводим  
❌ не редактируется вручную  

-------------------------------------------------
3. Принципиальное решение
-------------------------------------------------

Нарушение SLA:

— НЕ является ошибкой системы  
— является фактом  
— фиксируется отдельным событием  

SLA:

— не управляет жизненным циклом  
— не изменяет статус тикета  
— не влияет на workflow  

-------------------------------------------------
4. Типы SLA
-------------------------------------------------

В системе фиксируются три базовых SLA.

-------------------------------------------------
4.1 Response SLA (реакция)
-------------------------------------------------

Отражает скорость реакции на создание заявки.

Формула:

Response SLA =
min(
  ticket_created → ticket_assigned,
  ticket_created → ticket_started
)

Смысл:

— фиксирует момент, когда система "отреагировала"  
— не зависит от начала выполнения работ  

-------------------------------------------------
4.2 Resolution SLA (выполнение)
-------------------------------------------------

Отражает длительность выполнения работ.

Формула:

ticket_started → ticket_done

Важно:

Resolution SLA измеряет календарное время выполнения,
а не трудозатраты.

Это означает:

— не зависит от количества исполнителей;  
— не увеличивается при параллельной работе;  
— отражает длительность процесса.

Разделение:

— Resolution SLA = duration (процесс)  
— Labour Time = сумма сессий (см. ADR‑006)

Смысл:

Resolution SLA показывает скорость выполнения,
а не объём затраченных ресурсов.

-------------------------------------------------
4.3 Closure SLA (закрытие)
-------------------------------------------------

Отражает время административного закрытия.

Формула:

ticket_done → ticket_closed

Смысл:

— контроль диспетчерского контура  
— выявление “узких мест” после выполнения работ  

-------------------------------------------------
5. Источники данных (жёсткое правило)
-------------------------------------------------

Данные                Источник

ticket_created        событие  
ticket_assigned       событие  
ticket_started        событие  
ticket_done           событие  
ticket_closed         событие  
work sessions         ADR‑006  

Запрещено использовать:

❌ UI  
❌ ручные поля  
❌ внешние системы  

Audit и sessions — единственный источник истины.

-------------------------------------------------
6. Пороги SLA
-------------------------------------------------

Пороги SLA:

— задаются конфигурацией системы  
— управляются SuperAdmin / Admin  
— не хардкодятся  

Могут зависеть от:

— приоритета  
— типа заявки  
— категории  

-------------------------------------------------
7. Нарушение SLA (sla_breached)
-------------------------------------------------

При нарушении SLA создаётся событие:

sla_breached

-------------------------------------------------
7.1 Условия создания
-------------------------------------------------

— при превышении порога  
— один раз на тип SLA  
— независимо от дальнейших действий  

-------------------------------------------------
7.2 Структура события
-------------------------------------------------

ticket_id  
sla_type (response | resolution | closure)  
threshold  
actual_value  
breached_at  
current_state  
actor_context  
metadata  

-------------------------------------------------
8. Уведомления и policy
-------------------------------------------------

Принцип:

sla_breached = событие  
уведомления = policy  

Поведение:

— возможны повторные напоминания  
— прекращаются при:

  ✅ назначении  
  ✅ старте работ  
  ✅ закрытии SLA  

-------------------------------------------------
9. Использование SLA
-------------------------------------------------

✅ уведомления  
✅ аналитика  
✅ BI  
✅ отчёты  
✅ эскалации  

-------------------------------------------------
10. Ограничения SLA
-------------------------------------------------

❌ не менять статус  
❌ не запускать workflow  
❌ не корректировать данные  
❌ не выполнять автоматические действия  

SLA — это сигнал, не команда.

-------------------------------------------------
11. Reporting & Metrics
-------------------------------------------------

Метрики рассчитываются централизованно:

— только из audit + sessions  
— без бизнес‑логики  

Отчёты:

— read‑only  
— не влияют на систему  

-------------------------------------------------
Разделение типов времени
-------------------------------------------------

В системе используются три уровня времени:

1. Response Time
— время реакции системы

2. Resolution Time
— календарное время выполнения работ

3. Labour Time
— суммарные трудозатраты (человеко‑часы),
  рассчитываются по сессиям (ADR‑006)

Принцип:

— показатели не взаимозаменяемы  
— используются для разных бизнес‑целей  

-------------------------------------------------
Инварианты ADR‑007
-------------------------------------------------

— SLA всегда рассчитывается  
— SLA не редактируется  
— нарушения фиксируются событиями  
— SLA не влияет на статус  
— audit + sessions — единственный источник  

-------------------------------------------------
Осознанно исключено
-------------------------------------------------

— ручная корректировка SLA  
— изменение истории  
— SLA‑логика в UI  
— автоматическое исправление  

-------------------------------------------------
Связь с другими ADR
-------------------------------------------------

ADR‑003 — статус и жизненный цикл  
ADR‑004 — события  
ADR‑005 — workflow  
ADR‑006 — фактическое время  

-------------------------------------------------
Итог
-------------------------------------------------

Система разделяет:

— State (Ticket.status) — управление процессом  
— Time (Resolution SLA) — длительность выполнения  
— Activity (Sessions) — фактические трудозатраты  

Это позволяет:

✅ корректно считать OEE  
✅ считать загрузку персонала  
✅ строить отчёты и аналитику  
✅ масштабировать систему  

ADR‑007 является частью production‑контракта ZAS‑TRAC.
