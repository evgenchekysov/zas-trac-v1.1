ADR‑006: Work Sessions & Time Tracking
Статус: Accepted (production baseline, beta‑compatible)
Дата: 2026‑04‑27
Контекст: ZAS‑TRAC Backend

-------------------------------------------------
Контекст
-------------------------------------------------

В системе ZAS‑TRAC учёт фактического времени выполнения работ является ключевым элементом:

— для контроля выполнения заявок;
— для анализа загрузки исполнителей;
— для расчёта SLA и MTTR;
— для прозрачного audit и retrospective‑анализа.

Система поддерживает:

— прерывистое выполнение работ;
— паузы;
— параллельную работу нескольких исполнителей.

В ADR‑003 зафиксирован жизненный цикл тикета.
В ADR‑005 — workflow‑паттерн orchestration.

Данный документ фиксирует модель рабочих сессий и учёта времени.

-------------------------------------------------
Решение
-------------------------------------------------

Учёт времени осуществляется через Work Sessions.

Work Session — это интервал фактической работы конкретного пользователя над тикетом.

Учёт времени:

— дискретный;
— неизменяемый после фиксации;
— полностью управляется workflow‑слоем.

-------------------------------------------------
Базовые понятия
-------------------------------------------------

Work Session

Рабочая сессия:

— принадлежит одному пользователю;
— относится к одному тикету;
— имеет:
  — started_at
  — stopped_at;

— создаётся системой;
— закрывается системой;
— не редактируется после закрытия.

---

Активная сессия

Активная сессия — сессия без stopped_at.

Активная сессия возможна только при состоянии тикета IN_PROGRESS.

⚠️ Обратное не гарантируется:

Состояние IN_PROGRESS не требует наличия активной сессии
в каждый момент времени.

Это допустимо:

— между операциями workflow;
— при незавершённых пользовательских действиях;
— в промежуточных состояниях системы.

Таким образом:

— активная сессия требует IN_PROGRESS;
— IN_PROGRESS не гарантирует наличие активной сессии.

-------------------------------------------------
Совместная работа исполнителей (beta)
-------------------------------------------------

Система допускает параллельную работу нескольких исполнителей
над одним тикетом.

Разрешено:

✅ несколько исполнителей → несколько активных сессий  
✅ каждая сессия принадлежит своему пользователю  

Жёсткие ограничения:

❌ пользователь не может иметь более одной активной сессии в системе  
❌ overlap сессий одного пользователя запрещён  

-------------------------------------------------
Связь рабочих сессий и состояния тикета
-------------------------------------------------

Рабочие сессии и статус тикета отражают разные уровни системы.

Ticket.status:

— отражает стадию бизнес‑процесса;
— управляется через workflow;
— является коллективным состоянием.

Work Sessions:

— отражают фактическую работу пользователей;
— являются источником учёта времени;
— привязаны к пользователям.

Принцип разделения:

— status отвечает: «в каком состоянии находится заявка»
— sessions отвечают: «кто и сколько работает»

Связь:

— активные сессии возможны только в IN_PROGRESS;
— отсутствие активных сессий допускает состояние PAUSED;
— статус не является строго вычисляемым из сессий.

Допускается временное расхождение между:

— статусом тикета;
— фактической активностью пользователей.

Это является осознанным архитектурным решением.

-------------------------------------------------
Связь с жизненным циклом тикета (ADR‑003)
-------------------------------------------------

Состояние тикета   Активные сессии   Таймер

NEW                 ❌              ❌  
ASSIGNED            ❌              ❌  
IN_PROGRESS         ✅              ✅  
PAUSED              ❌              ⏸  
DONE                ❌              ⏹  
CLOSED              ❌              ⛔  

-------------------------------------------------
Создание рабочих сессий
-------------------------------------------------

Сессия создаётся только при переходе:

NEW → IN_PROGRESS  
ASSIGNED → IN_PROGRESS  
PAUSED → IN_PROGRESS  

Создание:

— выполняется в workflow;
— проверяет отсутствие активной сессии у пользователя;
— фиксируется событием work_session_started.

-------------------------------------------------
Остановка рабочих сессий
-------------------------------------------------

Активные сессии обязаны быть завершены при переходах:

IN_PROGRESS → PAUSED  
IN_PROGRESS → DONE  

Остановка:

— выполняется в workflow;
— фиксируется событием work_session_stopped.

-------------------------------------------------
Таймер выполнения тикета
-------------------------------------------------

В системе различаются два типа времени:

1. Время выполнения тикета (Downtime)

— определяется длительностью от начала IN_PROGRESS до завершения работ;
— используется для расчёта простоя оборудования (OEE);
— не зависит от количества исполнителей.

2. Трудозатраты (Labour Time)

— вычисляются как сумма длительностей всех рабочих сессий;
— увеличиваются при параллельной работе нескольких исполнителей.

Таким образом:

— downtime отражает влияние на оборудование;
— labour time отражает затраты человеческих ресурсов.

Один тикет может иметь:

— один интервал простоя;
— множество человеко‑часов.

-------------------------------------------------
Инварианты
-------------------------------------------------

— сессии создаются и закрываются только через workflow  
— активная сессия возможна только в IN_PROGRESS  
— пользователь имеет не более одной активной сессии  

— закрытая сессия неизменяема  
— удаление или редактирование сессий запрещено  

— время тикета (labour) вычисляется только из сессий  

Нарушение любого пункта считается архитектурной ошибкой.

-------------------------------------------------
События и audit (ADR‑004)
-------------------------------------------------

Каждое действие порождает событие:

— work_session_started  
— work_session_stopped  

События:

— фиксируются в audit;
— используются для:
  — восстановления истории;
  — расчёта SLA;
  — аналитики.

-------------------------------------------------
Архитектурное размещение
-------------------------------------------------

Workflow:

— создаёт и завершает сессии;
— проверяет инварианты;
— инициирует события.

Service:

— выполняет операции;
— не принимает решений о сценариях.

Repository:

— хранит данные;
— не содержит логики.

-------------------------------------------------
Осознанно исключено
-------------------------------------------------

— ручное редактирование времени  
— автоматическая коррекция статусов  
— overlap сессий одного пользователя  
— изменение истории  

-------------------------------------------------
Итог
-------------------------------------------------

Система разделяет:

— состояние процесса (Ticket.status)  
— фактическое выполнение (Work Sessions)  

Это обеспечивает:

— корректный учёт времени  
— поддержку multi-user сценариев  
— основу для SLA, OEE и аналитики  

ADR‑006 является частью production‑контракта ZAS‑TRAC.
