ADR‑006: Work Sessions & Time Tracking
Статус: Accepted (production baseline, beta‑compatible)
Дата: 2026‑04‑27
Контекст: ZAS‑TRAC Backend

Контекст
В системе ZAS‑TRAC учёт фактического времени выполнения работ является ключевым элементом:

для контроля выполнения заявок,
для анализа загрузки исполнителей,
для расчёта SLA и MTTR,
для прозрачного audit и retrospective‑анализа.

Система предусматривает:

прерывистое выполнение работ,
паузы,
участие нескольких исполнителей в одном тикете (beta‑сценарий).

В ADR‑003 был зафиксирован жизненный цикл тикета,
в ADR‑005 — workflow‑паттерн orchestration.
Необходимо зафиксировать модель рабочих сессий (work sessions) и правила учёта времени как архитектурный контракт.

Решение
В ZAS‑TRAC учёт времени осуществляется через рабочие сессии.

Work Session — это интервал фактической работы конкретного исполнителя над тикетом.

Учёт времени:

дискретный,
не редактируемый вручную,
полностью контролируется workflow‑слоем.


Базовые понятия
Work Session
Рабочая сессия:

принадлежит одному исполнителю
относится к одному тикету
имеет:

started_at
stopped_at


создаётся и закрывается системой
не редактируется после закрытия


Активная сессия

активная сессия — сессия без stopped_at
активная сессия возможна только при состоянии тикета IN_PROGRESS


Совместная работа исполнителей (beta‑поддержка)
В beta‑версии допускается параллельная работа нескольких исполнителей над одним тикетом.
Разрешённые сценарии

✅ несколько исполнителей → несколько активных сессий
✅ каждая сессия принадлежит своему исполнителю

Жёсткие ограничения

❌ один исполнитель не может иметь более одной активной сессии в рамках одного тикета
❌ overlap‑сессии одного исполнителя запрещены


Связь с жизненным циклом тикета (ADR‑003)

Состояние тикета   Активные сессии   Таймер
NEW                 ❌              ❌
ASSIGNED            ❌              ❌
IN_PROGRESS         ✅              ✅
PAUSED              ❌              ⏸
DONE                ❌              ⏹
CLOSED              ❌              ⛔

Создание рабочих сессий
Рабочая сессия создаётся исключительно при переходе тикета:
NEW → IN_PROGRESS
ASSIGNED → IN_PROGRESS
PAUSED → IN_PROGRESS

Создание сессии:

выполняется в workflow
проверяет отсутствие активной сессии у исполнителя
фиксируется событием work_session_started


Остановка рабочих сессий
Обязательная остановка активных сессий
Активные сессии обязаны быть остановлены при переходе тикета:
IN_PROGRESS → PAUSED
IN_PROGRESS → DONE


⚠️ Переход IN_PROGRESS → CLOSED запрещён согласно ADR‑003
и не рассматривается в модели сессий.

Остановка сессии:

выполняется в workflow
фиксируется событием work_session_stopped


Таймер выполнения тикета
Таймер тикета:

не является отдельной сущностью
вычисляется как:

сумма длительностей всех рабочих сессий всех исполнителей



Таким образом:

учитывается реальная трудоёмкость,
возможен расчёт человеко‑часов,
поддерживается корректная аналитика.


Инварианты (жёсткие правила)

Рабочая сессия создаётся и закрывается только через workflow
Активная сессия возможна только в состоянии IN_PROGRESS
У одного исполнителя может быть не более одной активной сессии на тикет
Закрытая сессия неизменяема
Удаление или редактирование сессий запрещено
Таймер тикета вычисляется только на основе сессий

Нарушение любого пункта считается архитектурной ошибкой.

События и audit (ADR‑004)
Каждая операция с рабочей сессией порождает событие:

work_session_started
work_session_stopped

События:

фиксируются в audit‑логе
используются для:

восстановления истории
расчёта SLA / MTTR
аналитики и отчётов




Архитектурное размещение ответственности
Workflow‑слой

управляет созданием и завершением сессий
проверяет инварианты
инициирует события

Service‑слой

реализует операции над сущностями сессий
не принимает решений когда начинать / останавливать

Repository‑слой

хранит данные сессий
не содержит бизнес‑логики


Что осознанно исключено
На текущем этапе:

ручное редактирование времени
автоматическая коррекция таймера
overlap‑сессии одного исполнителя
перерасчёт и изменение исторических сессий

Любые расширения допускаются только через новый ADR.
