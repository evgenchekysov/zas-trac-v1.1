ADR‑003: Ticket Lifecycle & Execution Model
Статус: Accepted (production baseline)
Дата: 2026‑04‑27
Контекст: ZAS‑TRAC Backend

Контекст
Тикет (заявка) является центральной доменной сущностью системы ZAS‑TRAC.
Через его жизненный цикл проходят основные операционные процессы: диспетчеризация, назначение исполнителей, выполнение работ, контроль времени, аудит и архивирование.
В ходе beta‑разработки сформировался реальный эксплуатационный процесс, включающий:

диспетчерскую очередь,
самозахват исполнителем,
административное назначение,
учёт активного времени,
временные паузы,
формальное закрытие после проверки.

Для предотвращения рассинхронизации между backend, UI, workflow и audit необходимо зафиксировать жизненный цикл тикета как архитектурный контракт.

Решение
В ZAS‑TRAC используется детерминированная state‑машина тикета со следующим набором состояний:
NEW
ASSIGNED
IN_PROGRESS
PAUSED
DONE
CLOSED

Состояние тикета отражает жизненный цикл обработки заявки, а не отдельные UI‑статусы.

Семантика состояний
NEW

Тикет создан и помещён в диспетчерскую очередь
Исполнение не начато
Назначение отсутствует
Отправляется уведомление в TG‑бот: «Новая заявка»


ASSIGNED

Тикет назначен диспетчером или администратором исполнителю или группе
Исполнение ещё не начато
Активной сессии нет
Таймер не идёт
Отправляется уведомление в TG‑бот: «Заявка назначена»


IN_PROGRESS

Исполнитель начал выполнение работ
Существует ровно одна активная сессия
Таймер исполнения запущен
Вход в состояние возможен:

через самозахват (NEW → IN_PROGRESS)
через принятие назначения (ASSIGNED → IN_PROGRESS)


Отправляется уведомление в TG‑бот: «Заявка в работе»


PAUSED

Исполнение временно приостановлено:

обед
ожидание запчастей
переключение на аварийные работы


Активная сессия закрыта
Таймер заморожен
Тикет не считается завершённым


DONE

Работы по тикету фактически завершены
Таймер исполнения окончательно остановлен
Тикет ожидает проверки и формального закрытия
Исполнитель больше не участвует в процессе


CLOSED

Тикет проверен и закрыт диспетчером / администратором
Состояние является терминальным
Любые изменения запрещены
Тикет переносится в архив


Допустимые переходы
NEW → ASSIGNED
NEW → IN_PROGRESS

ASSIGNED → IN_PROGRESS

IN_PROGRESS → PAUSED
PAUSED → IN_PROGRESS

IN_PROGRESS → DONE

DONE → CLOSED


Запрещённые переходы

NEW → DONE / CLOSED
ASSIGNED → DONE / CLOSED
PAUSED → DONE (без возврата в IN_PROGRESS)
DONE → IN_PROGRESS
CLOSED → любое состояние

Любой запрещённый переход считается архитектурной ошибкой.

Инварианты


Состояние тикета изменяется только через workflow

router‑слой не реализует логику переходов
service‑слой не принимает решений о допустимости перехода



У тикета существует ровно одно текущее состояние


IN_PROGRESS означает активное выполнение

допускается максимум одна активная сессия



PAUSED не является завершением работ

используется только для временной остановки таймера



DONE ≠ CLOSED

DONE — фактическое завершение работ
CLOSED — административная фиксация результата



CLOSED — терминальное состояние

никакие изменения, сессии и переходы невозможны




Таймер и сессии (связь с состояниями)

Таймер запускается исключительно при входе в IN_PROGRESS
Таймер приостанавливается при переходе в PAUSED
Таймер окончательно останавливается при переходе в DONE
У тикета может быть 0…N сессий, но не более одной активной

Детальная модель сессий и учёта времени описывается в отдельном ADR.

Архитектурное размещение ответственности


domain/

enum TicketState
знание допустимых переходов



services/

бизнес‑проверки
валидация сценариев



workflows/

orchestration переходов
создание сессий
генерация событий



routers/

HTTP‑контракты
не знают бизнес‑правил




Уведомления и аудит
Каждое изменение состояния:

порождает доменное событие
фиксируется в audit‑логе
может вызывать внешние уведомления (TG‑бот)

Контракт событий и audit‑модели описывается в отдельном ADR.

Осознанно исключено
На текущем этапе не допускаются:

автоматический re‑open тикетов
параллельные состояния
«мягкие» закрытия
скрытые статусные флаги вне state‑машины

Любое расширение жизненного цикла требует нового ADR.

ADR‑003 считается частью production‑архитектурного контракта ZAS‑TRAC.
