ADR‑003: Ticket Lifecycle & Execution Model
Статус: Accepted (production baseline)  
Дата: 2026‑04‑27  
Контекст: ZAS‑TRAC Backend  

-------------------------------------------------
Контекст
-------------------------------------------------

Тикет (заявка) является центральной доменной сущностью системы ZAS‑TRAC.

Жизненный цикл тикета отражает:
— обработку заявки;
— координацию исполнителей;
— контроль выполнения работ;
— административное закрытие.

В системе параллельно существует учёт рабочих сессий (ADR‑006),
однако статус тикета не является прямой функцией этих сессий.

Для предотвращения рассинхронизации между backend, UI, workflow и audit
жизненный цикл тикета фиксируется как архитектурный контракт.

-------------------------------------------------
Решение
-------------------------------------------------

В ZAS‑TRAC используется детерминированная state‑машина тикета:

NEW  
ASSIGNED  
IN_PROGRESS  
PAUSED  
DONE  
CLOSED  

Состояние тикета отражает стадию обработки заявки,
а не детальную активность исполнителей.

-------------------------------------------------
Семантика состояний
-------------------------------------------------

NEW

— заявка создана и помещена в очередь  
— исполнение не начато  
— участников может не быть  

---

ASSIGNED

— заявка назначена исполнителю или группе  
— работа ещё не начата  
— активных сессий нет  

---

IN_PROGRESS

— выполняются работы по заявке  
— в системе существует одна или несколько активных сессий  
  (не более одной на каждого исполнителя)  

Вход в состояние:

— NEW → IN_PROGRESS (самозахват)  
— ASSIGNED → IN_PROGRESS (принятие назначения)  

---

PAUSED

— работы временно остановлены  
— активных сессий нет  
— тикет остаётся в обработке  

---

DONE

— работы по заявке завершены  
— новые сессии запрещены  
— тикет ожидает проверки  

---

CLOSED

— заявка проверена и закрыта  
— состояние терминальное  
— любые изменения запрещены  

-------------------------------------------------
Допустимые переходы
-------------------------------------------------

NEW → ASSIGNED  
NEW → IN_PROGRESS  

ASSIGNED → IN_PROGRESS  

IN_PROGRESS → PAUSED  
PAUSED → IN_PROGRESS  

IN_PROGRESS → DONE  

DONE → CLOSED  

-------------------------------------------------
Запрещённые переходы
-------------------------------------------------

NEW → DONE / CLOSED  
ASSIGNED → DONE / CLOSED  
PAUSED → DONE (без возврата в IN_PROGRESS)  
DONE → IN_PROGRESS  
CLOSED → любое состояние  

-------------------------------------------------
Инварианты
-------------------------------------------------

— состояние тикета изменяется только через workflow  
— router не содержит бизнес‑логики  
— service не принимает решений о переходах  

— у тикета существует ровно одно состояние  

— IN_PROGRESS означает выполнение работ  
  (допускается ≥ 1 активная сессия)

— CLOSED — терминальное состояние  

-------------------------------------------------
Связь статуса тикета и рабочих сессий
-------------------------------------------------

Статус тикета и рабочие сессии представляют разные уровни системы.

Ticket.status:
— отражает стадию бизнес‑процесса  
— управляется через workflow  
— является коллективным состоянием  

Work Sessions (см. ADR‑006):
— отражают фактическую работу исполнителей  
— являются источником учёта времени  
— принадлежат пользователям  

Принцип разделения:

— статус отвечает на вопрос: «в какой стадии находится заявка»
— сессии отвечают на вопрос: «кто и сколько работает»

Связь между ними:

— наличие активных сессий допускается только в IN_PROGRESS  
— отсутствие активных сессий допускает переход в PAUSED  
— статус не является строго вычисляемым из сессий  

Допускается временное расхождение между:
— заявленным статусом  
— фактической активностью  

Данная модель является осознанным архитектурным решением.

-------------------------------------------------
Таймер и сессии
-------------------------------------------------

Таймер тикета не хранится отдельно и вычисляется как:

сумма длительностей всех рабочих сессий

Сессии:

— создаются в workflow  
— закрываются в workflow  
— не редактируются  

-------------------------------------------------
Уведомления и audit
-------------------------------------------------

Каждое изменение состояния:

— создаёт событие  
— фиксируется в audit  
— может инициировать уведомление  

-------------------------------------------------
Осознанно исключено
-------------------------------------------------

— автоматический пересчёт статуса из сессий  
— параллельные состояния  
— скрытые статусные поля  
— “мягкие” статусы  

Расширения допускаются только через новый ADR.

-------------------------------------------------
Итог
-------------------------------------------------

Модель системы разделяет:

— управление процессом (Ticket.status)  
— фактическое выполнение (Work Sessions)

ADR‑003 является основой жизненного цикла системы ZAS‑TRAC.
