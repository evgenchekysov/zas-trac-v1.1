## Статус документа
**Accepted (production)**  
Изменения допускаются только при осознанном пересмотре архитектуры.

---

## Цель

Зафиксировать архитектурный контракт системы ZAS‑TRAC в части:
- аутентификации,
- пользовательского контекста,
- ролей и распределения ответственности между слоями.

Данный контракт обязателен для всех новых endpoint’ов и сервисов.

## Базовый принцип

> **В систему всегда передаётся НЕ `user_id`, а пользовательский контекст.**

Идентификатор пользователя существует только как часть контекста.

## Role Model

В системе ZAS‑TRAC зафиксированы следующие роли:

- **SUPERADMIN** — управление ролями и правами пользователей
- **ADMIN** — диспетчер: подтверждение, закрытие и управление жизненным циклом заявок
- **EXECUTOR** — выполнение работ по заявкам
- **REQUESTER** — заявитель (гость)

### Invariant

> **Любая роль в системе имеет право создавать заявку.**

Создание заявки не является привилегией роли  
и не используется как механизм разграничения доступа.

Роль влияет **не на факт создания заявки**, а на:
- обработку,
- подтверждение,
- закрытие,
- управление правами других пользователей.

Permission Model
В системе ZAS‑TRAC используется модель прав (permissions) поверх ролевой модели.
Базовые определения

Роль — это организационная принадлежность пользователя
(шаблон, ярлык, стартовая конфигурация).
Право (permission) — это конкретная возможность выполнять действие в системе.
Роль ≠ Права.

Роль сама по себе не является источником полномочий и не используется
непосредственно для принятия бизнес‑решений.

Принципиальные положения

Все пользователи системы имеют право создавать заявки, независимо от роли.
Все ограничения применяются после создания заявки и основаны на правах.
Пользователь может обладать правами, выходящими за рамки его номинальной роли.
Набор прав пользователя определяется и изменяется суперадминистратором.


Матрица прав
Права представлены в виде атомарных флагов
(например, tickets.close, tickets.dispatch, reports.view).
Суперадминистратор управляет правами через матрицу разрешений
(интерфейс с галочками), где:

любому пользователю,
с любой ролью,
могут быть выданы или отозваны конкретные права.

Таким образом:

роль задаёт базовый ожидаемый профиль,
права определяют фактические возможности пользователя.


Примеры допустимых сценариев

Пользователь с ролью EXECUTOR может быть наделён правами:

анализа отчетов,
диспетчерской обработки заявок,
без изменения своей роли.


Исполнитель может временно выполнять функции администратора
(отпуск, болезнь, замещение) без повышения роли.
Суперадминистратор может тонко настраивать систему прав
без изменения кода и без добавления новых ролей.


Responsibility Split (обязательное правило)

Router:

извлекает пользовательский контекст,
не интерпретирует права,
не принимает решений.


Workflow:

передаёт контекст,
не содержит логики доступа.


Domain Service:

является единственным местом, где проверяются права
и принимаются решения.


Database / RLS:

ограничивает доступ к данным,
не реализует бизнес‑логику прав.

Инварианты

Проверка прав в router’ах и workflow запрещена.
Все бизнес‑решения на основе прав принимаются исключительно в domain‑service.
Любая логика доступа, реализованная вне доменного слоя,
считается нарушением архитектурного контракта.


Обоснование
Данная модель принята для:

поддержки реальных рабочих сценариев,
отказа от избыточного роста ролей,
безопасного делегирования полномочий,
и исключения beta‑технического долга при развитии системы.
