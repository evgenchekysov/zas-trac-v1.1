ADR‑005: Workflow Pattern & Orchestration Rules
Статус: Accepted (production baseline)
Дата: 2026‑04‑27
Контекст: ZAS‑TRAC Backend

Контекст
В ZAS‑TRAC основная бизнес‑ценность реализуется через последовательность действий:

создание и диспетчеризация тикетов,
назначение исполнителей,
запуск и остановку работ,
управление паузами и таймингом,
фиксацию событий и аудит.

К моменту beta‑версии сформировался отдельный workflow‑слой
(ticket_workflow.py, session_workflow.py), однако без формального архитектурного контракта:

бизнес‑логика рискует проникать в router и service‑слои,
правила изменения состояния могут быть нарушены,
события и таймеры могут вызываться неконсистентно.

Необходимо зафиксировать Workflow Pattern как обязательный и единственный способ orchestration бизнес‑процессов.

Решение
В ZAS‑TRAC используется явный workflow‑паттерн:

Workflow — это сценарий (use‑case), который оркестрирует бизнес‑процесс
и является единственным слоем, где разрешено изменять состояние тикета.

Workflow:

управляет переходами состояния (ADR‑003),
управляет созданием и остановкой сессий,
создаёт события (ADR‑004),
не зависит от HTTP / FastAPI,
не хранит собственного состояния.


Место workflow в архитектуре
Router (HTTP)
   ↓
Workflow (orchestration)
   ↓
Service(s) ──→ Repository (DB)
   ↓
State change + Event + Audit

Иного пути изменения состояния не существует.

Разграничение ответственности слоёв
Router‑слой (routers/)

принимает HTTP‑запрос
валидирует входные данные
определяет actor (пользователь / роль)
вызывает соответствующий workflow

Запрещено:

изменение state
вызов repository напрямую
генерация событий


Service‑слой (services/*_service.py)

инкапсулирует доменные операции
работает с repository‑слоем
реализует бизнес‑правила локального уровня

Service:

не знает сценария
не знает предыдущего и следующего состояния
не инициирует events

Допустимо:

CRUD‑операции
проверки согласованности данных


Workflow‑слой (services/*_workflow.py)
Workflow:

отражает один бизнес‑сценарий
вызывает один или несколько service
выполняет ровно один допустимый state‑переход
фиксирует событие
атомарен

Workflow — единственный слой, где разрешено:

менять состояние тикета
запускать / останавливать рабочие сессии
создавать audit‑event


Workflow‑сценарии в ZAS‑TRAC
На текущем этапе зафиксированы следующие workflow (фактически уже реализованы):
Ticket workflows

create_ticket_workflow
assign_ticket_workflow
start_ticket_workflow
pause_ticket_workflow
resume_ticket_workflow
complete_ticket_workflow
close_ticket_workflow

Session workflows

start_work_session_workflow
stop_work_session_workflow

Каждый workflow соответствует одному user‑action / system‑action
и не объединяет несколько независимых сценариев.

Инварианты workflow‑слоя

Любое изменение Ticket.state происходит только в workflow
Workflow не вызываются друг из друга

нет вложенности
нет цепочек


Workflow атомарен

либо выполнен целиком
либо не делает изменений


Workflow не хранит state

всё состояние хранится в DB


Workflow не зависит от FastAPI
Каждый workflow, изменяющий state, создаёт event

Нарушение любого пункта считается архитектурной ошибкой.

Связь с ADR‑003 (Lifecycle)


ADR‑003 определяет:

допустимые состояния
допустимые переходы



ADR‑005 определяет:

где и как эти переходы выполняются



Workflow обязан:

проверять допустимость перехода
явно указывать from_state → to_state
отклонять запрещённые сценарии


Связь с ADR‑004 (Events & Audit)
Каждый workflow, который:

меняет состояние тикета
открывает или закрывает сессию

обязан:

создать соответствующий event
зафиксировать audit‑запись

Workflow не обрабатывает внешние эффекты (TG, email),
а только инициирует событие.

Что workflow делать запрещено

❌ принимать HTTP‑запросы
❌ работать с FastAPI‑зависимостями
❌ хранить состояние между вызовами
❌ вызывать внешние сервисы напрямую (TG, webhook)
❌ содержать UI‑логику


Осознанно исключено
На текущем этапе:

нет saga / distributed workflow
нет вложенных или асинхронных workflow
нет state‑machine framework

Workflow реализован как явная, читаемая orchestration‑логика.

Последствия
✅ Бизнес‑процессы централизованы
✅ Архитектура проверяема
✅ Code review упрощается
✅ Новые сценарии добавляются без деградации системы
✅ Workflow‑слой становится частью production‑контракта

ADR‑005 является частью production‑архитектурного контракта ZAS‑TRAC.
