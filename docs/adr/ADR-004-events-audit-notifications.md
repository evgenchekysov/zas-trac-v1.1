ADR‑004: Events, Audit & Notifications Contract
Статус: Accepted (production baseline)
Дата: 2026‑04‑27
Контекст: ZAS‑TRAC Backend

Контекст
В системе ZAS‑TRAC значимая часть поведения не ограничивается изменением состояния тикетов, а включает:

фиксацию истории действий,
аудит ответственности,
внешние уведомления (Telegram‑бот),
подготовку данных для аналитики и SLA.

Без явного контракта событий:

логика рассеивается по коду,
audit становится неполным или недостоверным,
внешние интеграции (TG, отчёты) начинают влиять на бизнес‑процессы.

Необходимо зафиксировать единую модель событий (events) и правила их использования.

Решение
В ZAS‑TRAC вводится event‑ориентированный контракт, в котором:

Event — это неизменяемый факт, уже произошедший в системе
каждое значимое действие порождает событие
события используются для:

audit‑логирования
внешних уведомлений
аналитики


события не управляют бизнес‑логикой


Определение Event
Event — это запись, отражающая факт изменения состояния или выполнения действия, которая:

✅ фиксируется один раз
✅ не может быть изменена или удалена
✅ может быть потреблена несколькими подписчиками
❌ не принимает решений
❌ не инициирует изменение state

Event отвечает на вопрос:

«Что уже произошло?»


Типы событий
1. События жизненного цикла тикета
Полное соответствие ADR‑003:

ticket_created
ticket_assigned
ticket_started
ticket_paused
ticket_resumed
ticket_done
ticket_closed

Каждое событие соответствует одному факту перехода, а не действию пользователя.

2. События исполнения (execution)
Используются для фиксации рабочего времени:

work_session_started
work_session_stopped

Данные события не меняют состояние тикета напрямую, а отражают запуск или остановку рабочей сессии.

Момент генерации событий
Событие создаётся после успешного завершения действия:


ДействиеEvent
Создание тикетаticket_created
Назначение диспетчеромticket_assigned
Переход в IN_PROGRESSticket_started
Переход в PAUSEDticket_paused
Возврат в IN_PROGRESSticket_resumed
Переход в DONEticket_done
Переход в CLOSEDticket_closed
Если действие не завершилось успешно — событие не создаётся.

Источник событий (жёсткое правило)
✅ Только workflow‑слой имеет право создавать events
Запрещено:

❌ создание событий в router‑слое
❌ создание событий в domain‑моделях
❌ генерация событий из TG‑бота или внешних сервисов

Последовательность всегда следующая:

workflow проверяет допустимость действия
workflow изменяет состояние / создаёт сессию
workflow фиксирует событие


Audit‑модель
Каждое событие фиксируется в audit‑хранилище со следующим обязательным набором полей:

id
event_type
ticket_id
actor_id
actor_role
timestamp
from_state (если применимо)
to_state (если применимо)
metadata (JSON, расширяемо)

Инварианты audit‑лога

Audit‑лог иммутабелен
Записи не редактируются и не удаляются
Audit является источником истины для:

истории тикета
разборов инцидентов
SLA и отчётности




Уведомления (Telegram и другие)
Внешние уведомления реализуются как подписчики на события.
Архитектурная схема:
workflow → event → notification_service → TG‑bot

Принципы

TG‑бот не влияет на бизнес‑процесс
Ошибка отправки уведомления:

не отменяет событие
не откатывает действие


Один event может порождать:

0 или более уведомлений
разные каналы (TG, email, webhook)




Что события делать не должны
Событиям строго запрещено:

изменять состояние тикета
принимать бизнес‑решения
вызывать workflow
инициировать действия пользователя

Event — это фиксация факта, а не команда.

Связь с другими ADR

ADR‑003 — определяет, когда возникают события
ADR‑006 (планируется) — расширяет модель work‑session и таймера
ADR‑007 (планируется) — использует events для SLA и аналитики


Осознанно исключено
На текущем этапе:

не используется event‑sourcing
отсутствует replay событий для восстановления state
нет внешнего event‑bus

Event‑модель ориентирована на audit и интеграции, а не на управление состояниями.

ADR‑004 является частью production‑архитектурного контракта ZAS‑TRAC.
